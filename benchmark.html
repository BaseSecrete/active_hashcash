Notes

What this measures: End-to-end proof-of-work minting time in the browser. Hashes/sec normalizes for the random number of attempts needed, making it the most meaningful comparison across runs.

WebCrypto Batched — crypto.subtle.digest("SHA-256") inside a Web Worker. Nonces are batched via Promise.all to amortize async overhead. Batch size is configurable. This is the current PR #15 approach.

Pure JS — A synchronous SHA-256 implementation (~90 lines) in a Web Worker. Zero async overhead — tight loop. This is what Alexis (BaseSecrete) suggested for maximizing client-side speed.

WebCrypto Sequential — One await crypto.subtle.digest() per nonce, no batching. Similar to how Altcha does it (they compensate with multiple workers). Included as a baseline.

WASM (hash-wasm) — hash-wasm's hand-tuned WebAssembly SHA-256, loaded from CDN. Synchronous init()/update()/digest() loop in a Web Worker. The closest you can get to native C speed in a browser. Similar to how Cap uses compiled Rust→WASM for hashing.

WebGPU Compute — Uses navigator.gpu to run SHA-256 as a WGSL compute shader, dispatching batches of 256k nonces in parallel on the GPU. Supported in all major browsers: Chrome 113+, Edge 113+, Firefox 141+, Safari 26+. Extremely fast due to massive GPU parallelism, with some dispatch and readback overhead.

Dynamic batch size 2^(bits−2) — Alexis's suggestion. Scales with difficulty so bigger challenges use bigger batches.

To get a C reference speed, install hashcash and run:

hashcash -b20 -mS -r benchmark.example.com

Then enter the hashes/sec number in the Reference field above.

Why speed matters for security: The faster the JS client can mine, the smaller the advantage an attacker gets from using a compiled (C/Rust) implementation. Minimizing this ratio means the base difficulty (bits) can be set higher, improving protection against bots while keeping the UX acceptable for real users.
